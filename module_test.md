# Каталог модульных тест-кейсов (Критичные узлы системы)

> **Примечание:** В таблицах приведена унифицированная классификация типов сценариев: «Нормальный» (успешное выполнение), «Негативный» (корректная обработка ошибок) и «Граничный» (поведение на границах допустимых значений или особые структурные свойства).

---

## 1. Безопасность: Криптографический модуль API-ключей

**Модули**: `encryptUserApiKey` / `decryptUserApiKey`  
**Назначение**: Обеспечение безопасного хранения ключей пользователей от внешних LLM с использованием симметричного шифрования по стандарту AES-256-GCM.  

| ID | Входные данные (Операция) | Ожидаемый результат (Поведение системы) | Тип сценария |
|---|---|---|---|
| EK-01 | Передача корректного API-ключа и валидного 32-байтного ключа окружения для шифрования, а затем обратная расшифровка. | Цикл шифрования-расшифровки проходит успешно. Возвращается исходная строка ключа без потери данных. Зашифрованная строка имеет префикс версии формата `v1:`. | Нормальный |
| EK-02 | Попытка шифрования с использованием системного ключа недопустимой длины (не равного 32 байтам). | Функция прерывает работу и генерирует исключение, сообщающее о неверной длине ключа (защита от слабой криптографии). | Негативный |
| EK-03 | Отсутствие системного ключа шифрования (передан `null`). | Генерация исключения: `USER_API_KEY_ENCRYPTION_KEY is not configured`. | Негативный |
| EK-04 | Системный ключ шифрования передан как пустая строка. | Генерация исключения о неверной конфигурации сервера. | Негативный |
| EK-05 | Попытка расшифровать поврежденный шифротекст (имитация подмены данных злоумышленником путем изменения последних символов). | Генерация исключения криптографического ядра (сбой аутентификации GCM-тега), расшифровка отклонена. | Негативный |
| EK-06 | Попытка расшифровать пустое значение (передан `null` вместо зашифрованной строки). | Генерация исключения об отсутствии зашифрованных данных. | Негативный |
| EK-07 | Попытка расшифровать пустую строку. | Генерация исключения об отсутствии зашифрованных данных. | Негативный |
| EK-08 | Передача строки с неверным форматом разделителей (отсутствие блоков IV или AuthTag). | Генерация исключения о неверном формате шифротекста. | Негативный |
| EK-09 | Попытка расшифровки данных, зашифрованных неизвестной версией алгоритма (префикс `v2:` вместо ожидаемого `v1:`). | Генерация исключения о несовместимости версий формата. | Негативный |
| EK-10 | Два последовательных вызова функции шифрования для одного и того же API-ключа с использованием одного и того же системного ключа. | Возвращаются два абсолютно разных шифротекста (подтверждение уникальности генерируемого вектора инициализации — IV). | Граничный |

## 2. Разграничение доступа: Ролевая модель (RBAC)

**Модуль**: `requireRole` (Express Middleware)  
**Назначение**: Валидация прав доступа пользователей к маршрутам системы на основе их ролевой принадлежности (`student`, `author`, `admin`).

| ID | Входные данные (Контекст запроса) | Ожидаемый результат (Поведение системы) | Тип сценария |
|---|---|---|---|
| RR-01 | Пользователь с ролью `author` запрашивает доступ к маршруту, предназначенному только для авторов (`requireRole("author")`). | Доступ успешно предоставляется (вызывается функция `next()`), код ответа сервера не изменяется. | Нормальный |
| RR-02 | Пользователь с ролью `student` запрашивает доступ к закрытому маршруту авторов (`requireRole("author")`). | Доступ блокируется. Сервер возвращает HTTP-статус `403 Forbidden` с сообщением о нехватке прав. | Негативный |
| RR-03 | Запрос к защищенному маршруту выполняется неавторизованным пользователем (объект пользователя отсутствует в контексте). | Доступ блокируется. Сервер возвращает HTTP-статус `401 Unauthorized` с требованием предоставить JWT-токен. | Негативный |
| RR-04 | Пользователь с ролью `admin` обращается к маршруту, допускающему доступ сразу нескольким ролям (`student`, `author`, `admin`). | Доступ успешно предоставляется, так как роль пользователя присутствует в списке разрешенных. | Нормальный |
| RR-05 | Пользователь с ролью `admin` обращается к маршруту, строго ограниченному только для роли `author`. | Доступ блокируется (HTTP `403 Forbidden`). Демонстрация принципа наименьших привилегий: администратор не имеет неявного полного доступа. | Граничный |

## 3. Интеграция с ИИ: Безопасная передача контекста

**Модуль**: `getFailedTests`  
**Назначение**: Извлечение информации о не пройденных студентом тестах для передачи в LLM в качестве контекста, с обязательным маскированием секретных данных эталонных решений.

| ID | Входные данные (Результат выполнения кода) | Ожидаемый результат (Формируемый контекст для ИИ) | Тип сценария |
|---|---|---|---|
| GF-01 | На вход поступает объект с результатами тестирования, содержащий один открытый тест, который завершился с ошибкой (код возврата 1). | Возвращается массив с одним элементом, где корректно заполнены фактический и ожидаемый результаты, а тесту присвоен порядковый номер 1. | Нормальный |
| GF-02 | Объект результатов тестирования содержит только успешно пройденные тесты. | Возвращается пустой массив (ИИ не получает лишней информации, так как ошибок нет). | Нормальный |
| GF-03 | В функцию передан `null` вместо объекта с результатами. | Функция безопасно обрабатывает отсутствие данных и возвращает пустой массив (предотвращение ошибки `TypeError`). | Негативный |
| GF-04 | Объект передан, но свойство детализации тестов (`tests_result`) равно `null`. | Функция безопасно обрабатывает отсутствие детализации и возвращает пустой массив. | Негативный |
| GF-05 | Свойство детализации передано в неверном формате (строка вместо массива). | Данные игнорируются, возвращается пустой массив. | Негативный |
| GF-06 | Переданы данные о падении **скрытого теста** (с флагом `is_hidden: true`), содержащего секретные входные параметры и ожидаемый ответ. | Данные маскируются: поля `input`, `expected` и `actual` принудительно устанавливаются в `null`. ИИ получает информацию только о факте падения теста (предотвращение утечки эталонного решения). | Граничный |
| GF-07 | В результатах тестирования присутствуют сразу несколько упавших тестов. | Функция извлекает все упавшие тесты и корректно назначает им последовательную нумерацию (testNumber: 1, 2, ...), независимо от их исходного индекса в общем массиве. | Нормальный |
