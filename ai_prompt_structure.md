# Структура AI-запросов и формирование промпта (Yandex GPT)

В этом документе подробно разбирается архитектура отправки данных от клиентского приложения через Express-сервер в нейросеть Yandex GPT. Важно разделять **метаданные сервера (логи)** и **реальный массив сообщений (`prompt`)**, который поступает на вход языковой модели.

---

## 1. Двухуровневая структура данных при отправке

Когда пользователь отправляет сообщение в чат, данные проходят два уровня обработки:

```mermaid
graph TD
    Client[Клиент React] -->|POST /api/ai/chat| Server[Сервер Express / aiRoutes.js]
    
    subgraph ServerLog [Лог сервера / Метаданные]
        M1[userId, role, mode]
        M2[userInput - информационная копия]
    end
    
    Server --> ServerLog
    Server -->|buildPrompt()| PromptArray[Массив prompt]
    
    subgraph LLM_Payload [Только это уходит в Yandex GPT]
        P1[1. Системный промпт role: system]
        P2[2. Контекст урока role: user]
        P3[3. Контекст шагов role: user]
        P4[4. История чата role: user/assistant]
        P5[5. Текущий запрос userInput role: user]
    end
    
    PromptArray --> LLM_Payload
    LLM_Payload -->|HTTP Request| YandexGPT[Yandex GPT API]
```

## 2. Разбор структуры (что вы видите в логе консоли)

В консоли IDE при отправке запроса выводится следующий блок:

```json
{
  "interaction": "NEW",
  "userId": 1,
  "role": "author",
  "mode": "improve_text",
  "userInput": "@step1 Что стоит добавить в текущий текст. На что стоит обратить внимание?",
  "prompt": [ ... ]
}
```

### Уровень 1: Серверные метаданные (Внешняя оболочка)
Эти поля находятся в корне JSON-лога. **Они НЕ отправляются в Yandex GPT**, а используются исключительно сервером:
* `userId` (ID пользователя) — используется для проверки авторизации и получения зашифрованного API-ключа из базы данных PostgreSQL.
* `role` (`student` | `author` | `admin`) — определяет права доступа и набор доступных системных промптов.
* `mode` (например, `improve_text`, `fix_error`, `default`) — указывает функции `buildPrompt()` какой именно шаблон системного промпта выбрать.
* `userInput` — просто **информационная копия** текста, пришедшего с фронтенда. Выводится в лог для удобства отладки разработчиком.

---

### Уровень 2: Массив `prompt` (То, что отправляется в нейросеть)

В API Yandex GPT уходит **исключительно содержимое массива `"prompt"`**. Нейросеть читает этот массив последовательно (сверху вниз, от прошлого к настоящему).

Каждый элемент массива имеет строгую иерархию и назначение:

#### 1. Системный промпт (`role: "system"`) — *Базовый закон (Source of Truth)*
* **Расположение:** Всегда строго **первый элемент** массива `prompt`.
* **Назначение:** Задает личность, тон общения, формат вывода (Markdown) и границы дозволенного.
* **Правило важности:** Для языковых моделей роль `system` имеет наивысший приоритет. Инструкции внутри системного промпта перекрывают любые попытки пользователя изменить поведение модели (защита от *Prompt Injection*).

```json
{
  "role": "system",
  "text": "Ты профессиональный редактор. Всегда используй Markdown для форматирования ответов..."
}
```

#### 2. Контекст урока (`role: "user"`) — *Глобальное окружение*
* **Расположение:** Второй элемент (если контекст урока передан).
* **Назначение:** Сообщает нейросети общую тему урока и структуру шагов, чтобы модель понимала место текущей задачи в курсе.
* **Почему роль `user`:** Передача контекста от лица пользователя снижает его "инструктивный вес" по сравнению с `system`, предотвращая случайное восприятие текста лекции как команды для нейросети.

```json
{
  "role": "user",
  "text": "Контекст урока:\nАвтор редактирует урок: \"Введение в JavaScript\".\n\nУрок: Введение в JavaScript\nШаги:\n1. Зачем нужен JavaScript..."
}
```

#### 3. Контекст шагов (`role: "user"`) — *Локальное рабочее место*
* **Расположение:** Третий элемент (если открыт конкретный шаг).
* **Назначение:** Передает точные детали текущего шага: условие задачи (`Task`), написанный студентом код (`Student code`), ошибки компилятора (`stderr`), стандартный вывод (`stdout`) и упавшие тесты (`Failed tests`).
* **Важность:** Позволяет нейросети дать точный ответ, указывая на конкретную строчку кода студента или ошибку выполнения в Piston.

```json
{
  "role": "user",
  "text": "Контекст шагов:\n\n=== STEP 1 ===\nBlock ID: 1\nTitle: Зачем нужен JavaScript\nType: lecture\n\nTask:\nДобро пожаловать на наш курс!..."
}
```

#### 4. История чата (`role: "user"` / `"assistant"`) — *Память диалога*
* **Расположение:** Следующие от 0 до 10 элементов перед финальным вопросом.
* **Назначение:** Хронологический список предыдущих вопросов пользователя и ответов нейросети.
* **Безопасность:** Функция `normalizeMessages()` на сервере жестко отсекает любые манипуляции с историей на клиенте, оставляя только чистый текст реплик последних 10 сообщений.

```json
{
  "role": "user",
  "text": "Пробное сообщение для показа \"Контекста\" в \"Истории\"..."
},
{
  "role": "assistant",
  "text": "Ничего не предпринимаю."
}
```

#### 5. Запрос пользователя (`role: "user"`) — *Текущая задача (Триггер ответа)*
* **Расположение:** Всегда **самый последний элемент** массива `prompt`.
* **Назначение:** Свежий вопрос или команда от пользователя (`userInput`), ради которой и выполняется этот HTTP-запрос.
* **Почему он в самом конце:** Нейросеть воспринимает последний элемент массива как актуальную реплику собеседника. Прочитав сверху вниз правила (`system`), условия задачи (`контекст`) и прошлые реплики (`история`), модель генерирует ответ именно на это последнее сообщение.

```json
{
  "role": "user",
  "text": "@step1 Что стоит добавить в текущий текст. На что стоит обратить внимание?"
}
```

---

## 3. Резюме для разработчика

1. Поле `"userInput"` в верхней части лога консоли — это просто информационная строка для дебага, нейросеть её не видит.
2. Реально в Yandex GPT отправляется только массив `"prompt"`.
3. Нахождение вашего текущего вопроса в самом низу массива `prompt` (под историей и контекстами) — это **стандарт архитектуры LLM**, обеспечивающий точное понимание хронологии и контекста диалога.
